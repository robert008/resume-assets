Server級 AI 移植專案｜Flutter SDK化

On-device AI、Flutter Plugin 化、iOS/Android 跨平台（C++/FFI）

將原本在 Server AI辨識模組移植至flutter行動端，單張人臉運算約500毫秒體驗即時、不卡頓，並且封裝成 SDK，方便套用到任意fluter專案且無網也能辨識。

背景與目標

  • 主管希望能在無網的環境下也能使用臉部辨識功能，於是才有了這個將Server辨識AI移植到行動裝置的專案
  • 接手的時候原本的flutter工程師已經離職，從他的專案可以看到他試圖在flutter層引用 opencv 以及 onnxruntime框架想要直接在 flutter 層直接運算AI辨識
  • 在 flutter 層直接運算AI辨識運算的結果相當糟糕，且在無法使用 flutter 異步多工的狀況下，執行結果就是UI直接鎖死3~5秒鐘

我的角色

  • 團隊角色：影像辨識團隊由我與另一位Server(python)工程師共同負責
  • 由我主導：透過 AI 多次的資料搜查與確認，且與同事討論之後確定要使用 FFI 架構，由我同事完成 python 轉 C++的部分，剩下部分由我完成後續C++與flutter的介接

關鍵挑戰

  • 第一階段：成功使用FFI介面呼叫了C++程式
  • 第二階段：重新將專案建構成 flutter plugin 架構，且為了能讓外部可使用將C++程式碼封裝成動態庫
  • 第三階段：app應用端引用了 plugin 架構後遇到apple的巢狀式動態庫(Nested Dynamic Frameworks)問題，只好再將C++編譯成靜態庫，才能通過apple的程式碼檢驗
  • 坦白說，C++ 不是我的專長，尤其靜態庫的編譯/鏈結難度更是超出我的想像；很慶幸最終通過這個挑戰

技術架構

以下是完整的 Flutter FFI Plugin 三層架構設計，展示從 C++ 原始碼到 Flutter App 的整合流程：

Flutter FFI Plugin 三層架構：C++ → Flutter Plugin → Flutter App
  • C++ 層：ONNX Runtime + OpenCV 處理 AI 辨識邏輯
  • Plugin 層：FFI 介面橋接，提供統一的 Dart API
  • App 層：多個 Flutter App 可重複使用同一套 SDK
  • 靜態庫封裝：解決 Apple 巢狀動態庫限制，確保程式碼安全

開發一套完整的員工考勤管理系統，實現離線優先架構，確保無網路環境下仍可正常打卡並自動同步。整合人臉識別與 Beacon 技術達成精準考勤，採用MVVM 架構確保程式碼可維護性，並實現 401 錯誤自動處理與 Token 刷新機制。

背景與角色

關鍵挑戰

設計了網路感知式離線打卡系統，實現基於 Connectivity 監聽的智能模式切換機制，當檢測到離線狀態時自動啟動獨立的本地打卡流程，搭配離線數據持久化與增量同步策略，建立離線優先架構確保核心打卡功能在任何網路條件下都能穩定運作，有效解決弱網環境下的數據一致性問題。

企業級混合推播系統重構

針對公司既有 APNS 原生架構設計 Method Channel 雙向通信機制整合 Firebase FCM，克服 Flutter Firebase Plugin 與企業傳統推播服務的相容性限制，透過平台特定實作統一 iOS/Android 推播機制差異，實現原生推播與本地通知的無縫整合，確保跨平台通知顯示的一致性與穩定性。
